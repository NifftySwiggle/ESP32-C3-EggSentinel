What it detects

Threats it catches automatically

Egg Sentinel monitors your network continuously, using the same management-frame inspection techniques found in commercial Wi-Fi IDS appliances.

🔥

Deauth / Disassoc Flood

Sniffs unencrypted 802.11 management frames and flags floods — the classic sign of a Wi-Fi jamming or kick-everyone-off attack. Works against any tool sending deauth frames.

Always on
👻

Evil-Twin / Rogue AP

Periodically scans for access points broadcasting your exact SSID from an unrecognised MAC address — a classic honeypot technique to intercept your devices' traffic.

Always on
🎭

ARP Spoofing / MITM

Watches your gateway's MAC address. An unexpected change is a strong signal of ARP cache poisoning — someone telling your network "I'm the router now."

Always on
📡

New Device on LAN

Periodically ARP-scans your subnet and alerts when an unrecognised device joins. You'll know the MAC and IP within 60 seconds of them connecting.

Always on
🔵

BLE Advertisement Flood

Detects abnormal bursts of Bluetooth LE advertisements — the signal produced by "BLE spam" attacks that pop fake pairing popups on iPhones, Android devices, and Windows PCs nearby.

BLE toggle
👁️

New BLE Device Nearby

Tracks Bluetooth LE devices that appear in range and alerts on new ones. Note: devices with MAC randomization (most modern phones) will rotate addresses — expected behaviour, not a bug.

BLE toggle
⚡

Port-Scan Detection

Notices if something on your network is hammering the egg's own web server with rapid connection attempts — a sign that someone or something is actively scanning for open services.

Always on

How it works

From box to monitoring
in three minutes

No code editing, no SSH, no secrets in firmware. Everything is configured on the device itself.

1

Flash the firmware

Plug your ESP32-C3 "egg" board into USB, hit the flash button below, and the web flasher installs the firmware directly from your browser — Chrome or Edge only, no drivers or tools required. Make sure to set the partition scheme first (see Docs).

2

Join the setup hotspot

On first boot, the egg broadcasts its own Wi-Fi network called EggSentinel_Setup. Join it from your phone or laptop — a setup page pops up automatically (captive portal). Pick your home network, enter the password, optionally add a Telegram bot for alerts.

3

Monitor from the dashboard

The egg reboots, joins your network, and starts scanning. Open http://eggsentinel.local on any device for the live dashboard: threat log, LAN device list, real-time stats, Telegram settings, and a Bluetooth detection toggle — all persistent, no reflashing needed.

4

Get alerted — three ways

The built-in 0.42″ OLED flashes and shows the guard-dog mascot on any detection. The web dashboard updates in real-time. And if you've set up a Telegram bot, you'll get a push notification on your phone within seconds of anything suspicious.

Hardware required

Any ESP32-C3 "egg" board with a built-in 0.42″ SSD1306 OLED. These are widely available for a few pounds/dollars. The sketch is confirmed working with I²C on SDA=GPIO5 / SCL=GPIO6, which matches the most common egg board variant.

WiFi, WebServer, DNSServer, ESPmDNS, Preferences, WiFiClientSecure all ship with the ESP32 board package — nothing extra to install for those.

Board settings

Board: ESP32C3 Dev Module Partition Scheme: Minimal SPIFFS (1.9MB APP with OTA) ← important! see Partition Fix tab USB CDC On Boot: Enabled ← so Serial Monitor works over USB CPU Frequency: 160MHz (WiFi)

Why this matters

The default partition scheme ("Default 4MB with spiffs") gives the app only 1.2MB. With NimBLE added, the sketch needs ~1.5–1.9MB. The board has 4MB total flash — we just need to reallocate it.

⚠️ Skip this step and you'll get: Sketch too big (111%) even on a 4MB board.

How to fix it (takes 10 seconds)

  1. In Arduino IDE, open the Tools menu
  2. Find Partition Scheme
  3. Change it to "Minimal SPIFFS (1.9MB APP with OTA)"
  4. Upload again — done

Egg Sentinel uses Preferences (NVS) for settings storage, not SPIFFS — so no filesystem space is actually needed. The "Minimal SPIFFS" scheme gives the app ~1.9MB while keeping a small NVS partition for saved settings.

If that option isn't in your menu, alternatives that also work: "Huge APP (3MB No OTA)" or "No OTA (2MB APP/2MB SPIFFS)".

Create a Telegram bot (~3 minutes)

  1. Open Telegram and search for @BotFather
  2. Send /newbot and follow the prompts to pick a name
  3. BotFather sends you a token like 123456789:AAH4kf... — save it
  4. Search for @userinfobot, send any message — it replies with your chat ID (a number like 987654321)
  5. Message your new bot once first ("hi" is fine) — bots can't message you until you've messaged them

Add the credentials

Enter both values in the Egg Sentinel dashboard under Telegram alerts and hit Save. Click "Send test alert" to confirm it works. No reflashing ever needed to change Telegram settings.

You can also enter them during the initial Wi-Fi setup wizard if you have them ready.

What this can't do (important)

Being upfront about hardware constraints so you don't rely on this for something it can't actually do:

  • Can't see other devices' encrypted traffic. WPA2/WPA3 encrypts everything between devices and the router. Nothing can decrypt that without the key. This monitors network behaviour (who's on the network, what the gateway MAC is, whether management frames are flooding), not packet contents.
  • Can't fingerprint a specific attacking device. We can detect a deauth flood or a BLE advertisement flood, but there's no signature that says "this was a Flipper Zero specifically" vs any other tool doing the same thing. The attack pattern is what's detectable, not the device.
  • Can't see sub-GHz, NFC, IR, or RFID. These are completely different radio frequencies that a Wi-Fi/BLE chip has no hardware to receive. If a Flipper's using sub-GHz replay or NFC cloning nearby, this device has no way to know.
  • BLE "new device" alerts have false positives. Modern phones and earbuds rotate their Bluetooth MAC addresses every few minutes for privacy. Your own phone may occasionally trigger a "new BLE device" alert. This is normal, expected BLE behaviour, not a flaw in detection.
